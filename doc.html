UltraChamp
Codebase Documentation

UltraChamp (repo unitedchamp, live at ultrachamp.gr) is a Greek 5×5/6×6/7×7 mini-football league platform: a public site where fans browse teams, players, matches, standings and tournaments, plus an admin dashboard where organizers build tournaments (league, groups, knockout — including two-legged ties), record match results, and publish news. Built on Next.js 16 (App Router, RSC) with Supabase (Postgres + Auth + Storage) and Vercel KV for rate limiting. All UI text is Greek; route names are Greek transliterations (OMADES = teams, paiktes = players, anakoinoseis = announcements).

1What the app does

Two audiences, one Next.js app:

⚽ Public siteEditorial home with match calendar & video highlights · team registry & team pages · players directory with career/tournament stats · match explorer & match broadsheets · tournament viewers with live brackets & group tables · season standings · news (articles + announcements) · static info pages (rules, contact, policies).
🛠 Admin dashboardGate: Supabase role admin (full) or editor (content only). Tournament wizard — stages, groups, knockout brackets, fixture generation, two-legged ties · match results & per-player stats · postponements · team/player CRUD with logo/photo pipeline · articles/announcements CMS · user role management · stat-cache maintenance tools.
🧠 Domain enginePure fixture generators (round-robin, seeded knockout for any N) · a server-side progression engine that cascades a finished match into bracket advancement, group intake, standings recompute, next-stage seeding and stat-cache refresh · two-legged tie resolution decided on leg wins + penalties.
🧱 InfrastructureMiddleware (src/proxy.ts — Next 16's middleware) doing rate limiting, CSP nonces, security headers and auth gating · five Supabase client variants · signed-upload/proxy storage pipeline · GDPR consent-gated analytics · Greek-aware search normalization.

The repo also self-documents in docs/inventory/ (layer catalogs + 10 pipeline narratives, snapshot 2026-05-27). This page merges that inventory with a fresh 2026-07-04 exploration — the big deltas since May are two-legged knockout support ("Bracket 2.0"), the matches-v2 detail page promoted to live, and group-assignment orphan fixes.

2Architecture at a glance

Public page / UI Admin UI API route / middleware Server logic / lib Database / storage
Browser Public pages & dashboard
RSC pages under src/app/* (home, OMADES, OMADA, paiktes, matches, tournaments, standings, articles…) and the client-heavy admin under src/app/dashboard/*. Client state: one Zustand store for the tournament editor; everything else is local state + fetch.
▼
Edge Middleware — src/proxy.ts
Runs on every request: Vercel-KV rate limiting (global/IP/endpoint/auth/write/daily tiers) → CSP nonce → auth gate (/dashboard/* + all API mutations require session; role routing for admin vs editor) → security headers + CSP.
▼
Server ~50 API routes + server actions
src/app/api/* (auth, teams, players, matches, tournaments, stages, articles, announcements, storage, admin) and "use server" actions colocated with dashboard/match pages. Admin checks via app_metadata.roles.
▼
Lib Domain + plumbing — src/app/lib/*, TournamentCURD/*
Supabase clients (RSC / route / browser / admin service-role) · progression engine (progression.ts) · fixture generators (util/functions/*) · standings & bracket math · datetime (wall-clock convention) · rate-limit, CSRF, safe-redirect · image resolver · Greek search utils.
▼
Data Supabase Postgres + Storage · Vercel KV
~20 tables (tournament core, matches, teams/players, stats caches, content, discipline) + a handful of RPCs. Storage buckets for logos/photos/masks. KV holds rate-limit counters only. All result propagation is app code — no DB triggers.

6Known issues & cleanup map

Surfaced during mapping (many already tracked in docs/inventory/dead-ends.md and cleanup-candidates.md):

  • Layering inversion: /api/matches/[id] and the snapshot route import from dashboard/tournaments/TournamentCURD/ (progression engine + store types). The engine belongs in lib/.
  • Duplicate Supabase clients: Server.ts ≈ supabaseServer.ts; client.ts/supabaseClient.ts//api/me still on legacy auth-helpers.
  • Dual stat aggregates: legacy player_statistics and the new caches are both written — pick a canonical (that's why fix-stats exists).
  • Dual bracket-source representation on matches (stable round+pos vs resolved match ids) — intentional for reseed resilience, but two ways to say one thing.
  • Hardcoded values: bucket name GPSarchive's Project in ~10 files; OAuth BASE_URL; standings prize table; testimonials/feature cards on home.
  • Same-origin inconsistency: team/player/content routes add the (spoofable) request origin to the allowlist; match routes are strict.
  • Dead/typo files: TournamentCURD (sic), stages/leauge/, ModernKnockoutViewesr.tsx, LeaugeStage.tsx, * copy.tsx files, newEventPilll.tsx, /waves, /dotgrid, /hometest, classic home component set (only used by previews).
  • Duplicated home fetch logic between / and /preview/home-c (~200 lines) — partially consolidated into home/data.ts.
  • No committed baseline schema — core-table DDL exists only in production; export a 0000-baseline.sql.
  • Partial-failure risk on match finish: the progression cascade is many sequential writes with no transaction; a mid-cascade failure leaves standings/caches stale (reseed + refresh-stats are the recovery tools).
